# searxng — веб-поиск для ИИ-агентов через локальный SearXNG

| Чтобы дать агенту веб-поиск, обычно нужно… | …здесь |
|---|---|
| 🔑 завести аккаунт в поисковом API и достать ключ | ничего |
| 🙈 вставить ключ в `.env` (и постараться не закоммитить) | ничего |
| 🛠️ поднять сервер и разобраться с его конфигом | ничего |
| 🧩 включить формат ответа, обойти лимитер, выбрать движки | ничего |
| **итого действий:** | **скачать — и всё** |

При первом же поиске скилл сам поднимет локальный SearXNG в контейнере, сам
пропишет ему настройки (JSON API, лимитер, устойчивый набор движков) и
переиспользует его потом. Ни ключей, ни облака, ни ручной возни. 🎉

Скилл даёт агенту веб-поиск без API-ключей и без утечки запросов наружу: под
капотом — [SearXNG](https://docs.searxng.org) (метапоисковик, агрегирующий
Google, Brave, DuckDuckGo, Bing и 70+ движков), запущенный **локально в
контейнере**. Агент вызывает один скрипт и получает готовые ранжированные
результаты. Установка и запуск SearXNG происходят **автоматически при первом
использовании** — человек в процесс не вовлекается (кроме одного случая, см.
ниже).

> Этот README — на русском для удобства владельца. Сам скилл (`SKILL.md` и
> скрипты) — на английском.

## Что внутри

```
just-searxng-skill/              # репозиторий
├── README.md                    # этот файл — лендинг репо (установка, как устроено)
└── searxng/                     # САМ СКИЛЛ — это и копирует `npx skills`
    ├── SKILL.md                 # лёгкая инструкция для агента: как искать + примеры
    ├── ABOUT.md                 # что это и как устроено — для человека
    ├── config/
    │   └── settings.yml         # шаблон настроек SearXNG (включает JSON API, гасит лимитер)
    └── scripts/
        ├── search.sh            # главный вход: агент вызывает его для каждого поиска
        └── ensure-searxng.sh    # поднимает локальный SearXNG (docker/podman), идемпотентно
```

> Почему скилл лежит в подпапке, а не в корне репо: установщик `npx skills`
> считает `SKILL.md` в корне репозитория «однофайловым скиллом» и копирует
> **только** его, теряя `scripts/` и `config/`. Скилл в именованной подпапке
> копируется целиком — поэтому раскладка именно такая.

Философия (как и просили): `SKILL.md` — тонкий и заточен на «SearXNG уже готов,
вот как искать». Про установку там всего один абзац на самый крайний случай. Всю
чёрную работу (детект → запуск → ожидание готовности → понятная ошибка) делают
скрипты. Агент узнаёт детали установки, только если реально нечем поднять сервер.

## Как это работает

### `search.sh` — то, что вызывает агент

1. Разбирает запрос и опции.
2. Вызывает `ensure-searxng.sh`, чтобы гарантировать живой SearXNG (получает от
   него базовый URL).
3. Шлёт один запрос `GET /search?format=json` через `curl` (корректное
   url-кодирование, `User-Agent`, таймаут, проверка HTTP-кода).
4. Форматирует результат: `jq`, а если его нет — запасной парсер на `python3`.
5. Печатает ранжированные блоки `[n] title / url / snippet / [движки]`.

Режимы вывода: по умолчанию — текстовые блоки; `--urls-only` — только URL по
одному в строке (удобно скармливать в fetch); `--json` — сырой ответ SearXNG.

### `ensure-searxng.sh` — авто-подъём сервера (идемпотентный)

Порядок:

1. **Health-probe.** Реальный пробный JSON-запрос к `SEARXNG_URL`
   (по умолчанию `http://localhost:8888`). Если инстанс уже отвечает JSON — сразу
   успех. Проверяется именно JSON, а не просто «порт открыт», потому что
   HTML-only инстанс для нас бесполезен.
2. **Выбор рантайма.** `docker` (если демон жив) → иначе `podman` → иначе ошибка.
3. **Конфиг.** При первом запуске копирует `config/settings.yml` в рабочую папку
   `~/.cache/searxng-agent-skill/`, подставляя сгенерированный `secret_key`
   (через `openssl`, а если его нет — из `/dev/urandom`).
4. **Контейнер.** Если контейнер `searxng-agent-skill` существует — стартует его;
   иначе создаёт (`-p 127.0.0.1:8888:8080`, `--restart unless-stopped`, том с
   конфигом смонтирован в `/etc/searxng`). Слушает только на localhost.
5. **Ожидание готовности** до ~45с (поллинг JSON-probe).

Коды возврата: `0` — готов; `3` — нет рантайма (напечатана инструкция по
установке); `4` — рантайм есть, но сервер не поднялся/не отвечает JSON; `5` — нет
`curl`.

## Что делает наш `config/settings.yml`

Три вещи, из-за которых у большинства похожих скиллов поиск «молча не работает»
или часто отдаёт пусто:

1. **JSON API выключен по умолчанию.** Без `search.formats: [html, json]` любой
   запрос `format=json` возвращает HTML. Наш конфиг его включает.
2. **Бот-лимитер блокирует curl.** `server.limiter` для локального
   одно-пользовательского доступа выключен, иначе прилетает `403`.
3. **Устойчивый набор движков.** Дефолтные веб-движки SearXNG (brave, duckduckgo,
   google, startpage) агрессивно «подвешивают» сами себя при rate-limit/CAPTCHA,
   и когда все четыре падают одновременно — запрос возвращает 0. Мы **добавляем**
   к ним (не заменяя) три движка на независимых индексах без API-ключа — **bing,
   mojeek, presearch**, — которые почти не троттлятся, так что хоть один да
   ответит. Набор подобран эмпирически: остальные независимые кандидаты
   (yep, yahoo, qwant, mwmbl) в текущем образе оказались сломаны/заблокированы.

`use_default_settings: true` означает, что мы переопределяем только эти пункты
(а список `engines` **сливается** с дефолтным по имени движка), всё остальное
берётся из дефолтов SearXNG — устойчиво к обновлениям образа.

## Требования и первичная установка

В типичном случае нужен только **рабочий контейнерный рантайм** — `docker` или
`podman`. Всё остальное (образ SearXNG, конфиг, запуск) скрипт делает сам.

Если рантайма нет вообще, `ensure-searxng.sh` не трогает систему сам, а печатает
готовые команды под текущую ОС и выходит (так мы решили: без тихих установок и
sudo-сюрпризов). Разово выполнить одну из них:

- **macOS, Docker Desktop:** `brew install --cask docker`, затем открыть Docker.app
- **macOS, лёгкий CLI:** `brew install colima docker && colima start`
- **macOS, Podman:** `brew install podman && podman machine init && podman machine start`
- **Debian/Ubuntu:** `sudo apt-get install -y docker.io && sudo systemctl enable --now docker`
- **Fedora/RHEL:** `sudo dnf install -y podman`

Для парсинга желателен `jq` (иначе используется `python3`). Если нет ни того, ни
другого — скрипт попросит поставить `jq`.

## Установка самого скилла

Проще всего — через [`npx skills`](https://github.com/vercel-labs/skills)
(ставит скилл в любой поддерживаемый агент: Claude Code, Codex, Cursor и др.):

```bash
# глобально — доступен во всех проектах
npx skills add AleshaIdetRvat/just-searxng-skill -g

# или только в текущий проект
npx skills add AleshaIdetRvat/just-searxng-skill
```

Обновить позже — `npx skills update`, удалить — `npx skills remove searxng`.

Либо вручную — склонировать репозиторий и скопировать из него папку скилла
`searxng/` туда, где ваш агент ищет скиллы:

```bash
git clone https://github.com/AleshaIdetRvat/just-searxng-skill /tmp/jss
cp -r /tmp/jss/searxng ~/.claude/skills/searxng
chmod +x ~/.claude/skills/searxng/scripts/*.sh
```

## Настройка через переменные окружения (всё опционально)

| Переменная | Назначение | По умолчанию |
|---|---|---|
| `SEARXNG_URL` | использовать уже запущенный инстанс как есть (контейнер не поднимается) | `http://localhost:8888` |
| `SEARXNG_PORT` | порт хоста для контейнера | `8888` |
| `SEARXNG_IMAGE` | образ | `searxng/searxng:latest` |
| `SEARXNG_CONTAINER` | имя контейнера | `searxng-agent-skill` |
| `SEARXNG_SKILL_HOME` | папка состояния/конфига | `~/.cache/searxng-agent-skill` |
| `SEARXNG_VERBOSE` | `1` — печатать прогресс-логи (детект рантайма, запуск, ожидание). По умолчанию тёплый путь молчит, видны только ошибки и разовое уведомление о холодном старте | _(выкл)_ |

## Диагностика

- **«every queried engine was temporarily unavailable».** Бесплатные движки
  временами ловят rate-limit/CAPTCHA и на время «подвешиваются». Благодаря
  устойчивому набору (bing/mojeek/presearch) полный отказ теперь редок, но
  возможен при сильном бёрсте. Скрипт честно об этом сообщает. Решение: подождать
  ~30–60с и повторить, либо задать другие `--engines` / `--categories`.
- **HTTP 403/429.** Обычно это лимитер. У нашего локального конфига он выключен;
  если вы указали свой/удалённый инстанс — выключите `server.limiter` там.
- **Не поднимается контейнер.** Логи: `docker logs searxng-agent-skill`. Частые
  причины: занят порт `8888` (задайте `SEARXNG_PORT`), медленно тянется образ,
  или в вашем `settings.yml` не включён JSON.
- **Полный сброс:** `docker rm -f searxng-agent-skill && rm -rf ~/.cache/searxng-agent-skill`.

## Приватность и ресурсы

SearXNG крутится на `127.0.0.1`, наружу не торчит, запросы к внешним движкам идут
от вашей машины. Контейнер поднят с `--restart unless-stopped` — переживает
перезагрузку и остаётся «тёплым» для быстрых последующих поисков. Остановить:
`docker stop searxng-agent-skill`.

## Что позаимствовано у референсов

- Тонкий `SKILL.md` → один бинарник → сниппет-first выдача — из `ask-search`.
- Обязательное включение `search.formats: [html, json]` — единственное, что
  делал `searxng-3`, и чего не хватало почти всем остальным.
- `curl` вместо urllib (устойчивее к HTTP-нюансам), режим `--urls-only` для
  пайпа в fetch — из `ask-search`.
- Структурированные, «действенные» сообщения об ошибках вместо сырых трейсбеков.
- **Наше сверху:** реальный авто-детект + авто-запуск сервера (ни один референс
  этого не делал), явная обработка `unresponsive_engines` и эмпирически
  подобранный устойчивый набор движков (bing/mojeek/presearch) поверх дефолтов.
